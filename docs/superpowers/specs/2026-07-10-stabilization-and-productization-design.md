# 3D HOUSE FS 稳定化与产品化设计规格

- 日期：2026-07-10
- 状态：已批准；实现计划见根目录 `plan.md`
- 决策：采用“稳定化后产品化”路线，不继续以堆叠展示功能为首要目标
- 目标周期：单人约 6–8 周，按阶段门禁交付；新增周期主要用于 SketchUp 风格的 3D 编辑交互与跨设备验收

## 1. 背景与审查结论

当前仓库已经具备可运行的 Next.js 3D 工作台、共享户型模型、浏览器端热力/气流求解、风水参考分析和 FastAPI stub。类型检查、生产构建和现有后端测试可以通过，但项目仍属于功能丰富的原型，不具备稳定发布所需的安全边界、运行时数据校验、模拟回归测试和可维护模块边界。

现有 `plan.md` 已经过时：撤销/重做、浏览器保存、JSON 导出和专业罗盘等条目已经实现，而当前真正阻碍产品化的是以下问题：

1. 浏览器 AI Key 会随 `browserConfig` 发送到 Next.js API，与界面“只保存在浏览器”的表达不一致；用户可控 Base URL 还形成服务器侧请求风险。
2. AI 图片、提示词和分析 API 缺少严格的服务端规模限制，可能造成内存、CPU 或调用费用滥用。
3. 导入 JSON 主要依赖类型断言，没有版本、结构、唯一 ID、几何边界和数组规模校验。
4. 前端 physics-lite 与 Python 后端旧插值逻辑已经分叉，后端模型还缺少 `devices`。
5. 共墙拓扑通过完整墙段字符串推断，部分共墙可能误判为外墙，影响气流源汇。
6. Web Worker 没有使用 transferable，失败时会回退到主线程同步求解。
7. `ThreeSceneCanvas.tsx`、`AppShell.tsx` 和 `globals.css` 体积过大，并保留未使用的旧热力/气流渲染代码。
8. TypeScript 侧没有单元测试或 E2E 测试，lint 命令因缺少 ESLint 配置而失败。

## 2. 目标

本轮稳定化必须实现：

- 建立单一、版本化、可在运行时验证的 `HouseLayout` 数据契约。
- 使导入、编辑、AI 草案和 API 输入经过同一套验证与规范化流程。
- 修复 AI 调用的隐私表述、SSRF、上传规模和滥用风险。
- 为热力、气流和几何拓扑建立确定性测试夹具与可解释诊断。
- 把编辑状态、历史记录、场景图层和样式拆成可独立理解与测试的模块。
- 参考 SketchUp 的低学习成本交互，为现有户型数据模型提供可靠的 3D 选择、导航、吸附、尺寸输入和直接编辑体验。
- 建立 lint、单元测试、构建和关键用户流程的自动化门禁。
- 保留当前单层矩形户型和 physics-lite 产品定位，不夸大工程精度。

## 3. 非目标

本轮不实现：

- 工程审图级 Navier–Stokes CFD 或后端 GPU 求解。
- 多楼层、任意多边形房间、CAD/BIM 导入。
- 账号、支付、云项目同步和多人协作。
- 家具资产库、自动报价引擎或完整 PDF 排版系统。
- 对任意私有 OpenAI-compatible 地址提供无约束服务器代理。
- SketchUp 级任意面建模、布尔运算、自由曲面、组件市场和完整 Push/Pull 实体内核。

## 4. 总体架构

```text
UI 编辑命令 / 导入 JSON / AI 草案
              |
              v
packages/core: versioned schema -> validate -> normalize -> domain commands
              |
              +---------------------> local persistence / export
              |
              v
simulation worker: topology -> airflow -> coupled heat -> summaries
              |
              v
scene layers + analysis panels + renovation report
```

核心原则是：数据契约只有一个来源，所有入口先验证；求解与渲染彻底分离；安全边界按浏览器模式和服务器模式明确区分。

## 5. 数据契约与编辑器设计

### 5.1 版本化 Schema

在 `packages/core` 中使用 Zod 建立 `HouseLayoutSchema`，新增 `schemaVersion: 2`。旧存档在导入时通过显式迁移函数升级，导出始终写当前版本。

迁移边界固定如下：当前仓库产生的无 `schemaVersion` JSON 一律识别为 `v1`；显式 `schemaVersion: 1` 使用同一迁移器升级到 `v2`；仅支持 `v1 -> v2`。`schemaVersion > 2`、非整数版本、无法满足 v1 最低字段要求的历史对象直接拒绝，并返回“不支持的版本”或具体字段问题。P1 必须提供无版本现有存档、显式 v1、合法 v2、未知未来版本和损坏 v1 五类固定夹具。

必须验证：

- 所有数字有限且处于合理范围。
- 房间、墙、门窗、传感点、设备 ID 唯一。
- 房间尺寸为正且位于布局边界内。
- 房间发生正面积重叠时返回错误；仅接触边界允许。
- 门窗必须关联有效墙体并位于墙段范围内。
- 设备必须关联有效房间并位于房间范围内。
- 限制最大规模：32 个房间、256 段墙、256 个门窗、64 个传感点、64 个设备。

导入失败时返回结构化问题列表，不再把未知数组直接断言为 `HouseLayout`。删除墙体或房间时通过领域命令级联处理关联门窗和设备，避免孤立引用。

### 5.2 编辑命令与历史

保持 React 内置状态方案，不引入大型状态库。把布局变更集中为可测试的领域命令，例如：

- `moveRoom`
- `resizeRoom`
- `addCustomWall`
- `removeWall`
- `addOpening`
- `updateDevice`
- `replaceLayout`

历史记录保存命令结果，并将连续数字输入合并为一次历史事务，避免每个键入字符产生一条撤销记录。快捷键支持 `Ctrl/Cmd+Z`、`Ctrl/Cmd+Shift+Z` 和 `Ctrl/Cmd+S`。

墙体和其他实体统一使用无冲突 ID 生成器，不再依据当前数组长度生成 ID。

## 6. AI 安全边界

### 6.1 浏览器 Key 模式

浏览器 Key 模式直接从浏览器请求用户配置的 Provider。Key 只保存在浏览器并且不经过本项目服务器；若 Provider 不允许 CORS，界面应明确提示用户改用服务器模式，而不是静默代理。

### 6.2 服务器模式

服务器模式只允许使用环境变量配置的 Provider URL、模型和 Key。客户端不能覆盖服务器 Base URL 或请求路径。

服务器模式定位为受信任的单租户/私有部署能力，默认关闭。启用它必须同时配置 `AI_SERVER_ENABLED=true`、服务器密码和 Provider 配置。无账号的公开匿名服务器 AI 不在支持范围内；若部署到公网，必须由上游身份网关保护该接口，否则保持禁用。

服务器接口必须增加：

- 提示词最大 4,000 字符。
- 最多 3 张图片，每张最大 5 MB，仅接受允许的图片 MIME 类型。
- 请求体和响应体超时与大小限制。
- 单进程限制按可信客户端 IP + 路由计数：每 10 分钟最多 10 次生成、最多 2 个并发请求；全局最多 4 个并发 Provider 请求。超过限制返回 `429` 和 `Retry-After`。仅在显式 `AI_TRUST_PROXY=true` 时读取受控代理写入的首个 `X-Forwarded-For`；否则不信任任何转发 IP，并把请求归入一个更保守的全局未识别客户端桶。多实例部署必须由上游网关提供等价或更严格的共享限流。
- 不记录密码、API Key、图片 Data URL 或完整提示词。
- 统一的错误代码，响应中不暴露内部 Provider 地址和底层异常细节。

`/api/analyze` 若保留，也必须使用同一运行时 Schema 和输入规模限制；若没有实际消费者，则删除该未使用路由。

## 7. 模拟与拓扑设计

### 7.1 拓扑

增加独立的布局拓扑模块，将共线且重叠的墙段切分为规范化边，明确标注外墙、共墙和开口区间。气流源汇不得再通过完整墙段字符串相等来判断内外墙。

重叠房间、越界房间和孤立开口必须在进入 rasterize 之前被拒绝。模板和 AI 草案使用相同拓扑构建流程。

### 7.2 确定性与可信度

求解器继续定位为 physics-lite。P3 开始时先冻结下列小型矩形夹具和当前诊断快照，再实施拓扑修复；验收使用固定输入、固定迭代次数和固定容差：

- 双房间实墙/开门对照：相同热源下，开门方案的相邻房间平均温度响应至少为实墙方案的 2 倍。
- 单房间设备对照：开启空调后房间平均温度至少下降 0.1°C；厨房热源至少上升 0.1°C。
- 双外窗风向夹具：迎风窗记录为正入口，背风窗不记录为正入口，并形成相反压力趋势。
- 双房间内门夹具：内门连接两房间，但 `inlets` 中不得出现该内门位置。
- 速度投影后 `divergenceMean <= 0.02` 且 `divergenceMax <= 0.25`；若初始快照已优于该值，不允许回退超过 10%。
- 同一运行时重复求解时 TypedArray 逐元素绝对误差不超过 `1e-6`，舍入后的房间摘要必须完全一致。

诊断面板继续展示 residual、divergence、coverage 等指标，同时增加“physics-lite、非工程 CFD”的固定说明。

### 7.3 Worker 协议

Worker 消息增加协议版本和请求 ID。返回结果时使用 transferable 传递 TypedArray buffer，减少主线程复制。Worker 初始化或运行失败时显示降级状态；主线程同步回退只用于小模型并设置规模上限。

## 8. 前端模块边界

### 8.1 AppShell

把 `AppShell.tsx` 拆为：

- `useLayoutEditor`：布局、选择和领域命令。
- `useLayoutHistory`：撤销、重做和事务合并。
- `useAiDraft`：AI 配置、请求和草案状态。
- `useWorkspaceUi`：模式、图层、检查器和底部面板状态。
- `StudioTopbar`、`ToolRail`、`Inspector`、`AnalysisDock`：纯展示/事件组件。

### 8.2 3D 场景

把 `ThreeSceneCanvas.tsx` 拆为：

- `StructureLayer`
- `OpeningLayer`
- `DeviceLayer`
- `HeatLayer`
- `AirflowLayer`
- `CompassLayer`
- `BaguaLayer`
- `SceneControls`

删除未被渲染的旧 `HeatmapOverlay`、`AirflowOverlay` 及其工具函数。几何体、材质和纹理必须在依赖变化和卸载时释放。

当前屏幕坐标画墙不再伪装成三维地面绘制。推荐本轮采用明确的顶视正交编辑模式，通过射线与地面相交取得户型坐标；普通透视浏览模式不接收画墙输入。

### 8.3 SketchUp 风格的 3D 编辑交互

本轮参考 SketchUp 的交互语言，但只映射到当前单层、矩形房间、墙中心线、门窗和设备数据模型，不复制其完整实体建模内核。

导航与选择约定：

- 左键负责选择和当前工具操作；中键拖动环绕视角；`Shift + 中键` 平移；滚轮缩放。
- 提供顶视、等轴测、前视三个快捷视角，以及 `F` 聚焦选中对象。
- 普通透视模式通过 Three.js raycast 选择房间、墙、门窗和设备；选中、悬停和锁定状态必须有不同视觉反馈。
- 移动端使用单指选择/工具操作、双指缩放和平移；不要求在本轮实现复杂多指旋转手势。

工具与推断约定：

- 工具栏保留“选择、画墙、移动、门、窗、设备、测量”七类核心工具，使用一致的激活、取消和光标反馈。
- 画墙必须在明确的顶视正交工作平面上完成，通过 raycast 与地面相交取得世界坐标，不能继续按屏幕百分比映射坐标。
- 坐标契约固定为：布局 `x` 映射 Three.js 世界 X，布局 `y` 映射世界 Z，Three.js 世界 Y 是高度。红轴表示布局 X，绿轴表示布局 Y/世界 Z；蓝轴只用于视角和高度提示，本轮不能沿蓝轴移动房间、墙、门窗、设备或传感点，也不新增实体高度字段。
- 支持端点、中点、墙边、网格、平行和垂直推断。
- 桌面端 `Shift` 锁定当前推断；左/右方向键锁定红轴 X，上/下方向键锁定绿轴 Z；蓝轴不提供编辑锁定。`Esc` 取消当前操作；`Enter` 提交数值或完成操作。
- 开始移动或画墙后，可直接键入长度/位移数值并在状态栏/测量框显示，单位统一为米。
- 移动房间、设备或传感点时必须经过领域命令和边界校验；门窗沿所属墙段移动，不允许脱离墙体。
- 当前数据模型没有任意实体面，因此不实现真正 Push/Pull。界面不得出现会让用户误以为可自由拉伸任意面的控件。

交互状态栏持续显示当前工具、可用快捷键、推断类型、坐标/长度和校验错误。工具切换、模式切换和操作失败必须可撤销或无副作用。

工具行为与验收矩阵固定如下：

| 工具 | 可操作实体 | 核心行为 | 无效操作 |
|---|---|---|---|
| 选择 | 房间、墙、门窗、设备、传感点 | raycast 选中、悬停高亮、`F` 聚焦、`Delete` 删除允许删除的实体 | 点击空白清除选择；派生房间墙不可单独删除 |
| 画墙 | 自定义墙 | 顶视平面选起点/终点，端点/中点/网格吸附，轴锁定，键入长度后提交 | 长度小于 0.35m、越界或重复墙段时拒绝并保留可修改草稿 |
| 移动 | 房间、自定义墙、设备、传感点 | 拖动或键入 X/Z 位移；自定义墙作为整段平移；房间派生墙跟随房间更新 | 房间越界/重叠、实体脱离所属边界时拒绝并回到操作前状态 |
| 门 | 房间墙或自定义墙 | 点击墙上位置创建门，沿墙拖动 offset，可键入宽度和 offset | 未命中墙、超出墙长或与同墙开口重叠时拒绝 |
| 窗 | 房间墙或自定义墙 | 与门工具一致，并保留窗高/窗台高属性面板 | 未命中墙、越界或与同墙开口重叠时拒绝 |
| 设备 | 房间地面 | 选择设备类型后在房间内放置；后续用移动工具调整位置，用属性面板调整方向和强度 | 未命中房间或位置越界时拒绝 |
| 测量 | 地面点、端点、中点 | 两点测距，结果显示在状态栏和临时标注；`Esc` 清除 | 结果不写入 `HouseLayout`，不创建持久尺寸实体 |

移动房间时不允许直接拖动某一面派生墙来改变尺寸；尺寸变化继续使用属性面板或后续独立 Resize 工具，本轮不新增 Resize 工具。移动自定义墙时，其关联开口与墙一起移动并保持 offset。

触屏端提供可点击的红轴/绿轴/自由移动锁定按钮，以及操作中的数值输入框和确认/取消按钮，替代 `Shift`、方向键和直接键入。移动端不提供蓝轴编辑，也不要求通过软键盘复现桌面快捷键。

### 8.4 样式与加载

把全局 token、工作台布局、检查器、底部面板和分析组件样式拆成职责明确的 CSS 文件。3D 工作台采用动态加载并提供错误边界和加载占位。

## 9. Python 后端定位

Python 后端本轮保持实验性 stub，不作为前端模拟结果来源。需要：

- 补齐 `devices` 等当前布局字段和 Schema 版本。
- 删除或明确标记已经过时的 6×6 热力插值与简化气流结果，避免被误认为与前端等价。
- 使用共享 JSON 夹具做请求兼容测试。
- README 明确后端当前只用于未来报告、AI 网关或高精度求解扩展。

如果后端分析接口没有实际消费者，优先缩减而不是维持两套重复算法。

## 10. 质量门禁

建立以下命令并纳入 CI：

```powershell
npm.cmd run lint
npm.cmd run typecheck
npm.cmd run test
npm.cmd run build
python -m pytest backend/tests
```

测试分层：

- `packages/core`：Schema、迁移、ID、几何和领域命令。
- `packages/simulation`：拓扑、热场、流场、确定性和诊断阈值。
- `apps/web`：hooks、序列化、AI 请求边界和关键组件行为。
- Playwright：模板加载、编辑、撤销/重做、保存/恢复、导入拒绝、分析切换和 AI 本地回退。

仓库卫生包括：忽略并停止跟踪 `*.tsbuildinfo`、运行日志和临时 Playwright 产物；保留可复用测试脚本和文档。

## 11. 分阶段交付与门禁

### P0：工程基线（1–2 天）

- ESLint CLI、测试运行器、CI 和生成物清理。
- 记录当前首页 First Load JS、三个默认模板的 Worker 求解时间、主线程长任务和移动端关键流程基线；性能测量使用固定 Node/浏览器版本、生产构建和同一台参考机器，各场景预热一次后取 5 次中位数。
- 门禁：lint、typecheck、现有 build 和后端测试稳定通过。

### P1：数据完整性（3–5 天）

- 版本化 Schema、迁移、导入限制、唯一 ID、几何约束和领域命令。
- 门禁：非法/旧版/超大布局夹具均有确定结果，编辑不产生孤立引用。

### P2：AI/API 安全（3–5 天）

- 浏览器直连与服务器模式分离，上传/提示词限制、限流和错误模型。
- 门禁：浏览器 Key 不进入应用服务器；用户输入不能改变服务器 Provider 地址。

### P3：模拟可信度（5–8 天）

- 规范化共墙拓扑、求解器回归测试、Worker transferable 和降级状态。
- 门禁：所有基准夹具满足方向性、散度和确定性断言。

### P4：前端拆分、SketchUp 风格交互与性能（10–15 天）

- 拆分 AppShell、场景图层和 CSS，移除旧代码，加入错误边界与动态加载。
- 增加顶视工作平面、raycast 选择、SketchUp 风格视角控制、轴向推断、吸附、测量框和快捷键。

### P5：用户流程验收（3–5 天）

- 桌面 Playwright 覆盖模板加载、七类工具的核心成功路径、顶视画墙、数值定长、键盘轴锁定、视角切换、撤销/重做、保存/恢复、非法导入拒绝、三类分析切换和 AI 本地回退。

- 门禁：上述核心用户流程自动化通过且无未处理控制台错误；README、边界声明和新 `plan.md` 与代码一致。

## 12. 风险与取舍

- 浏览器直连 AI 受 Provider CORS 限制，但这是保证浏览器 Key 不经过应用服务器的明确取舍。
- 共墙拓扑重构会改变部分热力/气流结果，因此必须先建立回归夹具，再替换旧判断。
- 拆分大组件时不同时重做视觉设计，避免结构改造与审美改版互相放大风险。
- Python 后端暂不追求算法对齐，以避免在没有消费者和标定数据时维护第二套求解器。
- 性能目标先建立基线再优化，门禁关注可重复测量而不是主观流畅度。

## 13. 完成定义

本轮完成需要同时满足：

1. 数据入口经过统一 Schema，旧存档可迁移，非法输入可解释地拒绝。
2. AI Key、Provider 地址、上传和错误处理符合已声明的安全边界。
3. 模拟核心有自动化夹具，内外墙和内门拓扑行为正确。
4. Worker 使用 transferable，主线程回退受控。
5. 大型组件完成职责拆分，旧渲染代码被删除。
6. lint、typecheck、单元测试、构建、后端测试和核心 E2E 均纳入 CI。
7. README、规格和 `plan.md` 对当前能力、限制和下一阶段保持一致。
8. 七类 SketchUp 风格工具在桌面端具备成功/失败自动化用例；移动端具备选择、顶视画墙、屏幕轴锁定、屏幕数值输入和确认/取消用例，测量结果保持瞬态。
