# 3D HOUSE 平台规划方案

## 1. 项目目标

打造一个面向住宅空间分析的 Web 平台，支持以下核心能力：

- 用户以适合小白的方式自定义墙体、门、窗的位置、尺寸和数量，实时生成 3D 户型。
- 基于房屋结构、地理位置、朝向、天气与用户输入，输出热力图和房间气流图。
- 将风水规则参数化，输出完整信息、依据和可视化标注，并通过 3D 罗盘辅助用户自行分析。
- 为后续 AI 生成户型、AI 辅助编辑和智能分析预留统一数据接口。

该平台本质上是一个“参数化空间建模 + 环境模拟 + 规则分析 + AI 扩展”的复合系统，需要优先保证数据模型稳定、模块职责清晰、交互闭环完整。

## 2. 现有架构理解

当前架构已具备较清晰的分层思路：

- `apps/web`：Next.js 主应用，负责页面、3D 场景、控制面板、API Route 和状态管理。
- `packages/core`：核心逻辑层，适合承载几何生成、风水算法、共享类型。
- `packages/viewer`：独立 3D 渲染组件，适合封装 Three.js 场景能力。
- `packages/simulation`：模拟引擎层，适合承载 CFD、插值和热力计算。
- `backend`：Python FastAPI 服务，适合处理 CFD、风水 Python 实现、AI 模型服务。

整体方向合理，但要真正落地，关键不是先做“炫技功能”，而是先建立统一的空间数据模型和清晰的数据流。

## 3. 需求拆解与头脑风暴结论

### 3.1 用户自定义墙、窗、门

这是平台的第一核心能力，建议采用“参数化建模”而不是静态模型拼装。

同时，交互方式必须面向小白用户，而不是照搬 CAD、Blender 或专业 BIM 软件的复杂操作模式。

推荐方案：

- 由用户在“顶视图轻绘制 + 3D 实时预览”的混合编辑器中创建房间轮廓、墙体、门窗开洞。
- `packages/core/geometry` 根据结构化参数生成几何体。
- `packages/viewer` 负责渲染、拖拽、选中、吸附和尺寸反馈。

为什么优先做参数化：

- 后续热力模拟、风向分析、风水分宫都依赖几何与拓扑关系。
- AI 输出也更适合生成结构化布局，而不是直接生成 glTF 模型。
- 参数化模型更容易支持编辑、撤销、版本保存和规则校验。

为什么不用“纯 3D 自由绘制”：

- 小白用户难以处理视角、深度、吸附、对齐和尺寸控制。
- 专业建模式交互学习成本高，会明显拉高流失率。
- 对本项目来说，分析准确性比建模炫酷感更重要。

推荐的小白交互原则：

- 默认使用顶视图绘制，右侧实时显示 3D 结果。
- 用“画房间”“加墙”“放门”“放窗”这类动作按钮代替专业术语。
- 通过吸附、正交约束、自动对齐、默认尺寸减少用户思考成本。
- 允许用户直接输入“房间长宽”“窗户宽度”而不是必须拖拽到精准值。
- 提供户型模板、引导步骤和撤销重做，避免一次性暴露过多复杂能力。

建议的首版绘制流程：

1. 选择模板或从空白开始。
2. 画外轮廓或直接输入房间尺寸。
3. 补内墙，系统自动吸附和校正。
4. 点击墙体后添加门窗。
5. 在 3D 视图中查看效果并微调高度或位置。
6. 完成后进入热力图、气流图和风水信息分析。

### 3.2 AI 构造 3D 图

AI 应作为第二阶段增强能力，而不是第一阶段依赖项。

推荐路线：

- 第一阶段先定义统一 `HouseLayout` 数据结构。
- 第二阶段接入 `backend/ai`，支持文本生成布局草案。
- 第三阶段再扩展到户型图识别、自然语言编辑、智能建议。

原因：

- 如果没有标准数据结构，AI 接入后会和前端渲染、模拟分析严重耦合。
- AI 输出不稳定，必须落到可验证、可编辑的中间结构上。

### 3.3 热力图生成

热力图不建议只依赖几个温度点直接“脑补”，应采用分层方案：

- MVP：平面插值，基于用户输入点位生成近似热力图。
- V2：加入房屋朝向、门窗位置、房间功能、基础材料参数，修正插值结果。
- V3：接入 `backend/cfd`，用 CFD 模拟结果和实测点融合。

推荐原因：

- MVP 能快速展示价值，技术复杂度可控。
- 纯 CFD 成本高、反馈慢，不适合作为首版必选能力。
- 插值 + 规则修正 + CFD 增强是更稳妥的产品路径。

### 3.4 房间气流图

房间气流图应分为“室外主导风向输入”和“室内实际气流可视化”两层。

推荐方案：

- 用户提供位置或地址，系统获取天气 API 的实时风向与风速。
- 用户可手动覆盖风向，满足特殊场景分析。
- 室内气流基于门窗开口、朝向和开闭状态做简化模拟，并在房间内以箭头流线或方向场展示。

这样做的好处：

- 不把天气 API 当作室内气流真值。
- 同时兼顾真实环境数据和用户主观校正。
- 更贴近用户理解，因为用户最终需要的是“房间里风怎么走”，而不是只有一个室外风向数字。

### 3.5 房屋位置与朝向

这是风水分析和环境分析的共同基础，必须成为建模流程中的必填数据。

建议输入方式：

- 地址或经纬度：用于天气与区域环境数据。
- 大门朝向角度：用于风水山向、朝向和主导通风判断。
- 建造/装修年份：用于风水运盘分析。

### 3.6 风水信息程序化

风水模块不能一开始追求“大而全”，建议从“可解释、可验证、只提供信息不代替判断”的规则集切入。

推荐切入顺序：

- 基础规则层：朝向、门窗关系、明厅暗房、穿堂煞等可解释规则。
- 九宫映射层：将户型映射到九宫区域。
- 飞星计算层：基于坐向和年份生成飞星盘。
- 信息呈现层：输出规则名称、触发条件、对应区域、飞星组合、参考含义和可选调整方向。

推荐原因：

- 用户更容易理解“规则 + 依据 + 可视化标注”的价值。
- 复杂命理体系可逐步叠加，不会阻塞首版交付。

风水模块的产品边界：

- 提供完整输入信息：朝向、年份、宫位、飞星、门窗位置关系。
- 提供规则依据：为什么命中、命中了哪条规则、落在哪个区域。
- 提供中性说明：每项信息的常见解释和注意点。
- 不直接输出“这个房子好/不好”“必须怎么改”这类替用户下结论的话术。

建议增加 3D 罗盘展示层：

- 在 3D 场景中叠加可旋转的罗盘或方位环，明确正北、坐向、向首和门向。
- 将九宫分区、飞星信息和关键房间区域映射到罗盘对应方位上。
- 支持点击罗盘方位高亮空间区域，反向点击房间时同步定位罗盘扇区。
- 将罗盘作为“信息导航器”使用，帮助用户理解风水信息与实际空间的对应关系。

## 4. 推荐产品路线

建议采用“三阶段推进，一条数据主线”的建设路线。

### 路线 A：先做完整 3D，再补分析

优点：

- 演示效果强，视觉冲击好。

缺点：

- 容易陷入渲染细节，核心分析能力迟迟无法成型。
- 没有统一数据模型时，后续模拟和风水会反复返工。

### 路线 B：先做统一数据模型 + 基础编辑器 + 轻量分析

优点：

- 架构最稳，后续 AI、模拟、风水都能复用。
- 能较早形成“输入-分析-反馈”的完整闭环。

缺点：

- 首版视觉表现可能没有那么华丽。

### 路线 C：先做 AI 生成 demo 拉效果

优点：

- 传播性强，容易快速出概念样机。

缺点：

- 工程基础不足，结果不可控。
- 演示价值大于交付价值。

### 结论

推荐采用路线 B。

原因是该项目的真正护城河不在于单一 3D 展示或 AI 生图，而在于“结构化空间模型驱动多种分析能力”的平台化基础。

## 5. 目标架构补全建议

### 5.1 前端层

`apps/web` 建议进一步拆分：

- `app/`：页面与路由。
- `components/Scene/`：场景容器、选中状态、相机与灯光、网格与辅助线。
- `components/Editor/`：小白引导式绘制、墙体绘制、门窗编辑、属性面板。
- `components/Analysis/`：热力图层、气流图层、风水信息图层、3D 罗盘图层。
- `stores/`：布局状态、场景状态、分析状态、用户配置。
- `utils/serializers/`：布局存储与前后端数据转换。

建议补充两个关键交互区：

- `components/EditorWizard/`：分步骤引导用户完成户型录入。
- `components/Templates/`：常见户型模板和快速起步入口。

### 5.2 核心共享层

`packages/core` 建议拆成以下子域：

- `types/`：`HouseLayout`、`Room`、`Wall`、`Opening`、`Orientation`、`SensorPoint`、`AnalysisResult`。
- `geometry/`：墙体生成、门窗开洞、房间边界、九宫切分基础能力。
- `fengshui/`：规则引擎、九宫分区、飞星计算、罗盘方位映射。
- `validation/`：布局合法性校验、坐向必填校验、几何冲突检测。

### 5.3 模拟层

`packages/simulation` 建议分成双轨：

- `interp/`：快速插值算法，服务实时预览。
- `cfd/`：高精度模拟接口定义，服务后端计算对接。

### 5.4 后端层

`backend` 建议明确服务职责：

- `main.py`：统一 API 入口。
- `cfd/`：高精度热力/气流模拟任务。
- `fengshui/`：Python 版计算服务或规则校验补充。
- `ai/`：文本生成布局、图像转布局、自然语言编辑。

如果后续复杂度增大，可考虑把 `ai` 和 `cfd` 拆成独立微服务，但第一阶段不建议过早拆分部署。

## 6. 核心数据模型建议

整个项目必须先定义统一的中间数据模型，建议以 `HouseLayout` 为中心。

建议包含：

- 项目基础信息：名称、地址、经纬度、时区、建筑年份、装修年份。
- 空间结构：楼层、房间、多边形边界、墙体厚度、门窗开口。
- 朝向信息：大门朝向、房屋主朝向、窗户朝向。
- 材料信息：墙体、地面、玻璃、保温参数。
- 环境输入：室外天气、用户温度点、门窗开闭状态。
- 分析结果：热力图、房间气流图、风水宫位、飞星盘、罗盘方位映射、规则命中信息、参考说明。

关键原则：

- 所有前端编辑结果都先写入 `HouseLayout`。
- 所有模拟和 AI 输出也都落回 `HouseLayout` 或其派生结果。
- 3D 渲染只是 `HouseLayout` 的一种视图，不应成为数据源头。

## 7. 分阶段实施计划

## 阶段一：基础可用版（MVP）

目标：完成“自定义户型 + 基础分析闭环”。

交付内容：

- 建立 monorepo 基础目录和包边界。
- 定义 `HouseLayout`、`Wall`、`Door`、`Window` 等共享类型。
- 实现“小白友好”的顶视图绘制 + 3D 实时预览编辑器。
- 实现墙体、门窗的参数化建模与 3D 渲染。
- 支持基本编辑能力：新增、删除、移动、缩放、属性修改、吸附对齐、默认尺寸。
- 采集房屋位置、大门朝向、建造年份等关键输入。
- 实现简化热力插值可视化。
- 实现基础房间气流图可视化。
- 实现基础风水规则信息展示、可视化标注和 3D 罗盘叠加。
- 提供模板户型、引导流程和撤销重做。
- 支持布局保存、导入、导出。

验收标准：

- 用户可从零创建一个基础户型。
- 小白用户无需学习专业建模概念，也能完成基础户型绘制。
- 修改门窗与墙体后，场景实时更新。
- 输入若干温度点后可生成基础热力图。
- 输入位置、风向和门窗状态后可生成房间气流图。
- 输入朝向和年份后可生成完整的风水信息结果，并通过 3D 罗盘辅助查看，但不直接替用户下结论。

## 阶段二：增强分析版

目标：让分析从“静态规则”升级为“环境驱动”。

交付内容：

- 接入天气 API 获取实时风向、风速和温度。
- 实现外部风向与内部通风路径的简化推断。
- 引入材料参数和房间用途，优化热力分析权重。
- 实现九宫区域映射与飞星盘计算。
- 增强 3D 罗盘交互，支持宫位、方位、房间联动高亮。
- 增加分析报告页，输出热力、气流和风水信息总览。

验收标准：

- 用户输入地址后能自动补全环境参数。
- 热力图和气流图会随位置、朝向、门窗状态变化。
- 风水模块可展示结构化宫位结果、飞星信息、罗盘方位关系和规则依据。

## 阶段三：高级智能版

目标：引入高精度模拟和 AI 辅助生产能力。

交付内容：

- 接入 `backend/cfd` 执行高精度热力/气流模拟。
- 融合实测点与模拟结果，输出更可信的热力图。
- 接入 `backend/ai` 支持文本生成布局草案。
- 支持自然语言修改户型。
- 尝试户型图识别转结构化布局。

验收标准：

- 同一布局可切换“快速预估”和“高精度模拟”模式。
- AI 生成结果可被用户编辑、校验和分析。
- 分析流程不依赖手工重复录入数据。

## 8. 里程碑建议

### 里程碑 1：数据底座完成

- 完成共享类型设计。
- 打通前端编辑状态与序列化。
- 明确分析模块输入输出协议。

### 里程碑 2：编辑器可用

- 完成墙体/门窗创建与编辑。
- 实现场景交互、吸附、基础约束和小白引导流程。
- 可保存和恢复布局。

### 里程碑 3：基础分析闭环

- 完成热力插值展示。
- 完成基础房间气流图展示。
- 完成基础风水规则信息展示和 3D 罗盘展示。
- 完成分析图层与信息面板。

### 里程碑 4：环境联动

- 接入位置与天气。
- 完成风向数据接入和推断逻辑。
- 提供分析报告页。

### 里程碑 5：智能增强

- 接入 CFD 任务流。
- 接入 AI 布局生成和编辑。
- 形成完整的智能分析工作流。

## 9. 任务分工建议

如果是小团队推进，建议按能力域拆分而不是按页面拆分：

- 前端建模与交互：3D 场景、小白编辑器、状态管理、可视化图层、3D 罗盘交互。
- 核心算法与类型：共享类型、几何生成、规则校验、风水算法。
- 后端分析与服务：FastAPI、天气接入、CFD、AI 服务。
- 产品与验证：需求收敛、规则定义、样例户型、分析结果验收。

## 10. 关键风险与规避策略

### 风险 1：过早追求复杂 3D 表现

问题：

- 容易拖慢编辑器和分析核心能力建设。

建议：

- 首版先保证结构化正确和交互可用，再补材质、灯光和精细模型。

### 风险 2：没有统一数据模型

问题：

- AI、3D、风水、热力各做一套数据格式，后期整合成本极高。

建议：

- 第一优先级就是定义并冻结核心布局协议。

### 风险 3：风水信息表达越界

问题：

- 用户不容易信任黑盒结论，且平台若直接替用户下判断，容易引发误解。

建议：

- 所有输出都采用“规则来源 + 触发条件 + 对应位置 + 参考说明”的表达方式。

### 风险 4：模拟精度和性能冲突

问题：

- 高精度模拟无法满足实时交互。

建议：

- 前端用快速插值做即时反馈，后端用异步 CFD 做精算。

### 风险 5：AI 输出不稳定

问题：

- 生成内容可能不完整、不可施工、不可分析。

建议：

- AI 只输出草案，必须经过结构校验和用户确认后才能进入分析流程。

## 11. 验证与测试策略

建议从一开始就建立三类验证：

- 几何验证：墙体闭合、门窗是否越界、房间是否可识别。
- 规则验证：朝向、年份、风水规则命中逻辑是否正确。
- 分析验证：热力图和房间气流图结果与样例数据是否一致。
- 方位验证：3D 罗盘朝向、房屋坐向、九宫映射是否一致。
- 交互验证：首次使用用户能否在短时间内完成基本户型绘制。

建议测试方式：

- `packages/core`：单元测试，验证数据模型、几何算法、风水规则。
- `apps/web`：组件测试和交互测试，验证编辑器操作。
- `backend`：API 测试，验证天气、CFD、AI 接口输入输出。
- 端到端：使用固定示例户型验证建模到分析全链路。

## 12. 当前最优先落地清单

如果现在就开始执行，建议优先顺序如下：

1. 确定 `HouseLayout` 与核心类型。
